# H100/H200 混合環境對齊計畫

## Summary
把 repo 整理成適合 TWCC `H100/H200` 混合環境使用的 benchmark repo。
不再強調 `nano4` 或 `nano5` 差異，而是確保每次執行都能清楚記錄「在哪個 GPU、哪個節點、哪個 job 上跑」，並讓 `exp0`、`exp1` 都能一鍵到底產出可比較的結果。

## Key Changes
- 文件敘事改成「TWCC H100/H200 混合環境」
  - 頂層 README、`exp0` README、`exp1` README 不再把重點放在 `nano4` 或 `nano5`。
  - 文件明確說明目前機器可能分配到 `H100` 或 `H200`，因此每次 run 都必須記錄實際 GPU 型號。
  - `CMAKE_CUDA_ARCHITECTURES=90` 的說明改成 Hopper 家族通用，不再寫成只屬於 H100。

- 建立可追蹤的 run 命名與輸出結構
  - `exp0` 與 `exp1` runner 都改成每次執行建立獨立 run directory。
  - 建議格式：`results/exp0/run_<timestamp>_job<slurm_job_id_or_nojob>_<gpu_tag>/`
  - 範例：`results/exp0/run_20260410_1350_job16588_H200/exp0_seq.csv`
  - 若無 Slurm job id，使用 `nojob`；timestamp 一律保留，避免覆寫。
  - `gpu_tag` 由 runner 在執行當下偵測實際 GPU 產生，不從使用者手動輸入推定。

- 每次 run 產出一份 metadata 檔
  - 在 run directory 內新增 `run_meta.txt` 或 `run_meta.json`。
  - 至少記錄：timestamp、hostname、SLURM_JOB_ID、SLURM_JOB_PARTITION、CUDA_VISIBLE_DEVICES、實際 GPU 名稱、執行指令、git commit 或 dirty 狀態。
  - CSV 內已有 `device/sm/cc`，metadata 則補足 batch/節點層資訊，方便後續比對 H100 與 H200。

- 收斂 runner 與 batch wrapper 的責任
  - `scripts/run_exp0.sh` 與新增的 `scripts/run_exp1.sh` 作為正式 benchmark runner。
  - runner 專責：build、建立 run directory、偵測 GPU、命名輸出、執行 benchmark、寫 metadata。
  - Slurm 提交腳本不再和 benchmark runner 混在一起；若保留 batch 腳本，應改成明確的 wrapper 名稱或移到獨立位置，內容只負責載入模組與呼叫 runner。
  - README 只保留一套正式入口，避免混淆哪份腳本才是正確用法。

- `exp0` runner 對齊完整實驗定義
  - `exp0` runner 預設一次跑完 `seq`、`masked`、`gather` 三個 mode。
  - 三份 CSV 都放在同一個 run directory 內，便於同次 run 的橫向比較。
  - runner 額外把這次 run 實際落在哪種 GPU 明確寫出，避免只靠事後打開 CSV 才知道。

- `exp1` runner 改成真正 end-to-end
  - 新增 `scripts/run_exp1.sh`，讓 `exp1` 也有一致的 build/run/output 流程。
  - 由於目前 `exp1` 程式內部已自行配置並初始化 planes，不依賴外部 dataset，因此 runner 不需要額外準備資料檔。
  - runner 需在執行前輸出或記錄預估配置量，讓使用者知道 `n` 與 `plane_bytes` 會吃多少 device memory。
  - 若未來 `exp1` 改成要讀真實 encoded dataset，那會是新的 interface 變更，需另開一輪設計，不混在本次計畫。

- 結果與暫存檔管理
  - `.gitignore` 忽略 `build/`、`results/`、Slurm `*.log`/`*.err`、根目錄暫存 CSV。
  - 不把單次 benchmark 產物散落在 repo 根目錄。
  - 若要保留代表性結果，應從 `results/` 挑選後再用明確方式整理，不作為 runner 預設行為。

## Public Interfaces / Usage Changes
- 正式執行入口
  - `scripts/run_exp0.sh`
  - `scripts/run_exp1.sh`
- 正式輸出結構
  - `results/exp0/run_<timestamp>_job<id>_<gpu>/...`
  - `results/exp1/run_<timestamp>_job<id>_<gpu>/...`
- 正式對外保證
  - 每次 run 都能從目錄名與 metadata 直接看出是 `H100` 還是 `H200`
  - 每次 run 都不會覆寫前一次結果
  - `exp1` runner 在目前程式版本下是 self-contained、可一鍵到底

## Test Plan
- Runner 命名與追蹤
  - 連續執行兩次 `exp0` 或 `exp1`，結果不覆寫，且各自有獨立 run directory。
  - 在 Slurm 環境下，run directory 名稱包含 job id；在非 Slurm 環境下，名稱包含 `nojob`。
  - metadata 與 CSV 中的 GPU 名稱一致，能清楚分辨 `H100` 與 `H200`。

- `exp0` runner 驗證
  - 一次執行後同一 run directory 內出現 `seq`、`masked`、`gather` 三份 CSV。
  - 三份 CSV 都含 `device/sm/cc` 欄位。
  - README 描述的輸出位置、檔名與實際結果一致。

- `exp1` runner 驗證
  - runner 可從 build 到 run 直接完成，不需要任何外部資料檔。
  - 執行前會記錄或顯示 memory setup 資訊，執行時成功完成 plane allocation、pointer setup、output buffer allocation。
  - 成功產出 `exp1` CSV 與 metadata，且 CSV 中包含 `device/sm/cc`。
  - 若 `n` 設太大導致配置失敗，錯誤訊息要能明確指出是 setup/allocation 階段失敗，而不是安靜中止。

- Repo 清潔度
  - benchmark 跑完後，`git status` 不再被 `build/`、log、根目錄 CSV 汙染。

## Assumptions
- 主要需求不是綁定 `nano4` 或 `nano5`，而是支援 TWCC 上可能落到的 `H100/H200` 節點。
- 預設以 Hopper 家族為主，因此 `90` 或 `native` 都可作為文件與 runner 的建議選項，但 runner 不依賴使用者手動判斷 GPU。
- 目前 `exp1` 是 scan microbenchmark，不包含外部資料讀取與真實 FP64-to-byte-plane encoding pipeline；它的 runner 以內部 memory initialization 為準。
- batch 提交腳本與 benchmark runner 應分層，不再讓兩者都叫 `run_exp0.sh` 且各自做不同事。
